# Split `json_version_core` into a separately-published npm package

**Date:** 2026-07-16
**Status:** Approved design — pending implementation plan

## Goal

Today `crates/json_version_core` is a Rust crate whose prebuilt `.node` binaries are
bundled into `packages/json-version-plugin` at publish time. The crate's
`package.json` is `"private": true` and gets nothing done.

We want to publish the native module as a **standalone npm package**
(`@frada/json-version-core`), so other consumers (not just the Vite plugin) can
`require()` it directly. The plugin becomes a thin Vite wrapper that depends on
the published core via `workspace:*`.

Trigger for this refactor: adding
`"prepublishOnly": "napi pre-publish -t npm --no-gh-release"` to the crate's
`package.json`. That script only does anything if the crate is being published —
and that single line cascades into the restructure below.

## Constraints

- **Two packages, one version.** `@frada/json-version-core` and
  `@frada/json-version-plugin` always share the same version number, matched to
  the git tag (`v0.2.0` etc.).
- **Prebuilds bundled into the npm tarball.** `napi pre-publish -t npm
  --no-gh-release` puts all four platform binaries into the published artifact.
  No GitHub Releases for binaries.
- **Source-of-truth = `Cargo.toml`.** Version is bumped there first, then synced
  to the two `package.json` files via a tiny repo script (`scripts/sync-version.mjs`).
- **CI is the gate.** Both publish jobs verify `package.json` version equals the
  git tag, so drift fails loudly.

### `packages/json-version-plugin/src/native.ts` (after)

```ts
import { createRequire } from 'node:module';

const nodeRequire = createRequire(import.meta.url);

export interface ComputeOptionsNative {
  root: string;
  include: string[];
  length: number;
}

export interface NativeModule {
  computeVersion(opts: ComputeOptionsNative): string;
}

let _core: NativeModule | undefined;

/** Test-only seam — see __tests__/setup.ts. */
export function __setNativeForTests(mod: NativeModule) {
  _core = mod;
}

export async function loadNative(): Promise<NativeModule> {
  if (_core) return _core;
  const core = nodeRequire('@frada/json-version-core');
  if (typeof core.computeVersion !== 'function') {
    throw new Error(
      'json-version-plugin: @frada/json-version-core did not export computeVersion'
    );
  }
  _core = core;
  return _core;
}
```

Removed: `NativeLoadError`, `findBinary()`, `nativeDir()`, `moduleDir()`, the
linux/musl file-candidate fallback, and the `--no-js --dts native.d.ts` parallel
build that produced an empty `index.d.ts` in the crate.

## Version sync — `scripts/sync-version.mjs` (new)

```js
import { readFileSync, writeFileSync } from 'node:fs';

const cargoToml = readFileSync('crates/json_version_core/Cargo.toml', 'utf8');
const v = cargoToml.match(/^version = "([^"]+)"/m)[1];

const targets = [
  'crates/json_version_core/package.json',
  'packages/json-version-plugin/package.json',
];

for (const path of targets) {
  const json = JSON.parse(readFileSync(path, 'utf8'));
  json.version = v;
  writeFileSync(path, JSON.stringify(json, null, 2) + '\n');
}
console.log(`synced version ${v}`);
```

Add a root `package.json` script: `"version:sync": "node scripts/sync-version.mjs"`.
Documented in README as the first step of any release.

## CI changes — `.github/workflows/release.yml`

### Job graph

```
build (matrix: 4 legs)   — produces 4 .node binaries
        │
        ▼
publish-core           — needs: build
        │                downloads 4 binaries into crates/.../prebuilds/
        │                napi pre-publish bundles them into the npm tarball
        │                pnpm publish (publishConfig.access:public + prepublishOnly)
        ▼
publish-plugin         — needs: publish-core
                          pnpm install now resolves @frada/json-version-core
                          tsup build
                          pnpm publish
```

### `build` job (changed)

Only the upload step changes so each leg uploads its binary into a per-leg
subdir. `merge-multiple: true` in `publish-core` will then flatten them into
`crates/json_version_core/prebuilds/`.

```yaml
- uses: actions/upload-artifact@v4
  with:
    name: native-${{ matrix.suffix }}
-    path: ${{ runner.temp }}/napi-out/
+    path: ${{ runner.temp }}/napi-out/${{ matrix.suffix }}/
     if-no-files-found: error
     retention-days: 14
```

### `publish-core` job (new, replaces today’s `publish`)

Per-leg `actions/download-artifact@v4` with `merge-multiple: true` into
`crates/json_version_core/prebuilds/`. Verify all four `.node` files present
and non-empty. Verify package.json version equals git tag. Run `pnpm publish`
inside `crates/json_version_core` — `prepublishOnly` triggers
`napi pre-publish -t npm --no-gh-release`, bundling the prebuilds into the
tarball.

### `publish-plugin` job (new)

Same version-vs-tag check. `pnpm install --frozen-lockfile` happens **after**
`publish-core` completes so the workspace resolves `@frada/json-version-core`
to the freshly-published version. `pnpm tsup build` then `pnpm publish`.

Both publish jobs share the gating `if:` from today — push to `v*` tag, or
manual dispatch with `dry_run != 'true'`. Setting `dry_run=true` (or being on a
non-tag push) skips BOTH publishes, preserving today's "build only" mode but
now across the matrix.

## Test-suite hook change — `__tests__/setup.ts`

Drop the `NATIVE_BINARY_PATH` env-var path. Replace with a top-level call to
`__setNativeForTests(stub)` (imported from `../src/native.ts`) so
`loadNative()` returns the injected fake instead of touching the require cache.
This removes the need for a hand-crafted `.node` stub file entirely.

## Out of scope

- Adding a second Rust crate (no root `Cargo.toml` workspace changes).
- Switching to Changesets / release-please — three files is below the
  complexity threshold where tooling earns its keep.
- Dropping the `--no-gh-release` flag to use GitHub Releases for prebuilds —
  chosen by the user; bundling in the tarball is the model here.
- Splitting the existing `index.d.ts` content out of the JS plugin — currently
  the file in the crate is empty and the plugin's `types.ts` is the source of
  truth for the TS-side `PluginOptions`. After the split, only the core
  package's auto-generated `.d.ts` (function signature of `computeVersion`)
  matters for the public JS surface; the plugin's types stay where they are.

## Migration order (what code lands first)

1. Bump versions and add the new files (`scripts/sync-version.mjs`).
2. Update `crates/json_version_core/package.json` to the new shape and make
   `pnpm --filter @frada/json-version-core build` produce a working
   `index.js + index.d.ts` next to the crate.
3. Rewrite `packages/json-version-plugin/src/native.ts` and update its
   `package.json` to depend on the workspace crate. `pnpm --filter
   @frada/json-version-plugin build` + `pnpm --filter
   @frada/json-version-plugin test` should pass locally.
4. Update `.gitignore`.
5. Split `release.yml` into two publish jobs. Verify on a `dry_run` first.
6. First real release: bump Cargo.toml, run `pnpm run version:sync`, push tag
   `v0.2.0`, watch CI.
